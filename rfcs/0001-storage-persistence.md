# RFC 0001: Storage и персистентность состояния

**Статус**: Отклонено  
**Дата**: 2026-02-23  
**Автор**: Обсуждение в парном программировании

## Контекст

Plugin trait имеет метод `id()` с документацией "Unique ID that is used to store plugin progress", что предполагало реализацию механизма персистентности для состояния плагинов.

Изначальная задача была в High Priority списке TASKS.md.

## Проблема

Плагины могут хотеть хранить состояние между перезапусками бота:
- Счётчики (message count, reaction count)
- Last processed message ID
- Кэши (user info, channel metadata)
- Состояние обработки (для Layer 3 thread-bot)

## Рассмотренные варианты

### Вариант 1: Storage как параметр Plugin методов
```rust
async fn on_start(&self, config: &Arc<Configuration>, storage: &Arc<dyn Storage>) -> Result<()>;
```
**Минусы**: Ломает существующий API, breaking change для всех плагинов.

### Вариант 2: Storage в Configuration
```rust
pub struct Configuration {
    pub storage: Option<Arc<dyn Storage>>,
}
```
**Минусы**: Configuration из autogenerated mattermost-api, не можем модифицировать.

### Вариант 3: Storage внутри плагина (self-managed)
```rust
struct MyPlugin {
    storage: Arc<dyn Storage>,
}
```
**Плюсы**: Гибкость, не ломает API.
**Минусы**: Каждый плагин дублирует код инициализации.

### Вариант 4: TypedStorage wrapper
```rust
pub struct TypedStorage<S: Storage> {
    inner: Arc<S>,
    plugin_id: &'static str,
}
```
**Плюсы**: Type-safety, удобство использования, per-plugin namespace.

## Решение: НЕ реализовывать Storage в Layer 2

### Причины отказа

1. **Реконсиляция неизбежна**: При старте бота всё равно нужно реконсилировать состояние per-thread
   - События приходят через WebSocket в реальном времени
   - Пропущенные события при рестарте нужно догонять из Mattermost API
   - Невозможно просто "продолжить с последнего ID"
   - Реконсиляция происходит per-thread, не глобально

2. **Layer 3 сам управляет персистентностью**: 
   - thread-bot (Layer 3) реализует свою систему хранения тредов
   - Знает специфику: треды, сообщения, вызовы инструментов, metadata
   - Требует вторичные индексы для поиска ("долго не было ответа")
   - Layer 2 не должен диктовать как хранить сложные данные

3. **Простые use cases не требуют персистентности**:
   - Счётчики можно пересчитать при старте
   - Кэши можно заполнить заново из API
   - "Last processed ID" не решает проблему пропущенных событий

4. **YAGNI принцип**: 
   - Нет реального use case для Storage в Layer 2
   - Если понадобится - можно добавить позже (через middleware или в плагине)
   - Плагины, которым нужна персистентность, реализуют её сами

### Архитектурное обоснование

```
Layer 4 (Support Bot):    Бизнес-логика + LLM
         ↓
Layer 3 (thread-bot):     Управление тредами + СВОЯ персистентность + реконсиляция
         ↓
Layer 2 (mattermost-bot): Event streaming БЕЗ персистентности
         ↓
Layer 1 (mattermost-api): HTTP + WebSocket API
```

**Layer 2 (mattermost-bot)** остаётся простым event streaming фреймворком:
- Подключение к WebSocket
- Парсинг событий
- Распределение по плагинам
- Lifecycle hooks (`on_start`, `on_shutdown`)
- Middleware (IgnoreSelf)

**Layer 3 (thread-bot)** реализует свою персистентность:
- Специфичная для тредов структура данных
- Вторичные индексы
- Реконсиляция per-thread при старте
- Не зависит от generic Storage из Layer 2

## Возможное будущее

Если появится реальный use case для Storage в Layer 2:
- Можно реализовать как отдельный middleware
- Можно добавить optional Storage через Bot builder
- Можно оставить как ответственность конкретных плагинов

На данный момент - явно не нужно.

## Цитата из обсуждения

> "Казалось бы нам надо уметь начинать читать с определенного места, но мы ловим события и все равно следующему слою придется реконсилировать на старте все. Этого прям вообще не избежать. Причем реконсилировать per-thread"

Это ключевое наблюдение, которое привело к решению не реализовывать Storage в Layer 2.
