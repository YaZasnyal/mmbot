# CLAUDE.md - Руководство для работы с mmbot

## Обзор проекта

**mmbot** - это фреймворк для создания ботов для Mattermost, написанный на Rust. Проект имеет многоуровневую архитектуру, где каждый слой предоставляет API для более высокого уровня абстракции.

## Модель разработки

### Парное программирование с AI

Этот проект разрабатывается в модели **парного программирования**:

- **AI (Claude)**: Быстро набирает код, реализует задачи
- **Человек**: Проводит code review, принимает архитектурные решения

### Принципы работы

1. **Никакого vibe-кодинга**: Все изменения структурированы и продуманы
2. **Одна задача за раз**: Для качественного review работаем последовательно
3. **Небольшие MR**: Каждое изменение должно легко reviewиться
4. **Список задач в TASKS.md**: Все улучшения фиксируются и приоритизируются
5. **Review обязателен**: Код не мержится без человеческого review

### Workflow

```
1. Определить задачу из TASKS.md
2. AI реализует решение в отдельной ветке
3. Создать коммит с понятным сообщением
4. Человек проводит review
5. Если нужно - исправления
6. Merge в main
7. Обновить статус в TASKS.md
8. Перейти к следующей задаче
```

### Ожидания от AI

- Писать чистый, идиоматичный Rust код
- Следовать архитектуре проекта
- Документировать изменения
- Объяснять принятые решения
- Предлагать альтернативы при необходимости
- **Запускать `cargo fmt` перед коммитом**
- **Проверять код через `cargo clippy -- -D warnings`**

### Ожидания от человека

- Проводить вдумчивый review
- Задавать вопросы о неясных решениях
- Принимать финальные архитектурные решения
- Обновлять приоритеты задач

### Качество кода

Перед каждым коммитом обязательно:

1. **Форматирование**: `cargo fmt` - весь код должен соответствовать стандартному Rust стилю
2. **Линтинг**: `cargo clippy -- -D warnings` - все предупреждения clippy должны быть исправлены
3. **Сборка**: `cargo build` - проект должен собираться без ошибок

**Примечание**: Автогенерированный код в `lib/mattermost-api` исключён из проверок clippy (в нём заглушены все линты).

## Архитектура проекта

Проект организован как Cargo workspace со следующей структурой:

```
mmbot/
├── lib/
│   ├── mattermost-api/      # Нижний слой: автогенерированный API клиент
│   ├── bool_parser/          # Утилита для парсинга булевых значений из Mattermost API
│   └── mattermost-bot/       # Средний слой: базовый фреймворк для ботов
├── mmbot/                    # Верхний слой: конкретная реализация бота
└── notes/                    # Документация и заметки

```

### Слой 1: lib/mattermost-api

**Назначение**: Низкоуровневый API клиент для Mattermost

**Особенности**:
- Автоматически сгенерирован из OpenAPI схемы Mattermost (v4.0.0)
- Предоставляет типизированный доступ ко всем endpoints Mattermost API
- Использует `reqwest` для HTTP запросов
- Содержит модели данных для всех сущностей Mattermost (Post, Channel, User, Reaction и т.д.)

**Зависимости**:
- `reqwest` - HTTP клиент
- `serde` / `serde_json` - сериализация
- `bool_parser` - кастомный парсер для булевых полей (Mattermost иногда отдает строки вместо bool)

**Важно**: Этот слой генерируется автоматически. См. `/notes/generate_mm_api.md` для деталей регенерации.

### Слой 2: lib/mattermost-bot

**Назначение**: Минимальный фреймворк для создания любого бота

**Основные компоненты**:

#### `Bot` структура
Главный компонент, управляющий жизненным циклом бота:
- Подключается к Mattermost через WebSocket API
- Управляет переподключениями (автоматический retry с задержкой в 1 сек)
- Аутентифицируется с помощью bearer token
- Распределяет события между плагинами

#### `Plugin` trait
Интерфейс для создания обработчиков событий:

```rust
#[async_trait]
pub trait Plugin: Send + Sync + 'static {
    /// Уникальный ID плагина (для хранения прогресса)
    fn id(&self) -> &'static str;
    
    /// Быстрый фильтр событий (синхронный)
    fn filter(&self, event: &Arc<Event>) -> bool {
        true
    }

    /// Обработка события (асинхронная)
    async fn process_event(&self, event: &Arc<Event>, config: &Arc<Configuration>);
}
```

#### Типы событий (EventType)
Текущие поддерживаемые события:
- `Posted` - новое сообщение в канале
- `ReactionAdded` - добавлена реакция
- `ReactionRemoved` - удалена реакция

**Архитектурные решения**:
- События обрабатываются параллельно для каждого плагина
- Фильтр вызывается синхронно для быстрой проверки
- Обработка асинхронная, позволяет длительные операции
- События передаются через `Arc` для эффективности

**Файлы**:
- `lib.rs` - основная логика Bot
- `plugin.rs` - trait Plugin и Event
- `types.rs` - типы событий (EventType, PostedEvent, и т.д.)
- `nested_decoder.rs` - вспомогательный декодер для вложенных JSON структур

### Слой 3: lib/thread-bot (планируется)

**Назначение**: Упрощение работы с тредами в каналах

**Планируемые функции**:
- Реагирование на треды в каналах
- Сериализация обработчиков для одного треда (избегание race conditions)
- Прерывание старых обработчиков при новых сообщениях в треде
- Управление состоянием треда

**Интерфейс** (будет спроектирован позже):
Должен предоставить более высокоуровневый API для работы с тредами, чем просто обработка событий `Posted`.

### Слой 4: mmbot/

**Назначение**: Конкретная реализация бота

**Текущее состояние**: Содержит экспериментальный код для интеграции с:
- OpenAI API (локальный LLM через LM Studio)
- SurrealDB (для хранения данных)
- Обработка tool calls

**Примечание**: Основной код находится в `src/main.rs`, большая часть закомментирована.

## Утилиты

### lib/bool_parser

**Назначение**: Парсинг булевых значений из Mattermost API

**Проблема**: Mattermost API иногда возвращает булевые значения как строки `"true"`/`"false"` вместо JSON boolean.

**Решение**: Предоставляет serde deserializer'ы:
- `deserialize_bool` - для обязательных полей
- `deserialize_option_bool` - для опциональных полей

**Использование** в моделях:
```rust
#[serde(deserialize_with = "bool_parser::deserialize_option_bool")]
pub some_field: Option<bool>,
```

## Workflow разработки

### Добавление нового типа события

1. Добавить вариант в `EventType` в `lib/mattermost-bot/src/types.rs`
2. Создать соответствующую структуру события (например, `NewEventEvent`)
3. Добавить serde атрибуты для десериализации
4. Плагины автоматически смогут обрабатывать новое событие

### Создание нового плагина

1. Импортировать необходимые типы:
```rust
use mattermost_bot::{async_trait, Event, Plugin};
use mattermost_api::apis::configuration::Configuration;
```

2. Реализовать trait Plugin:
```rust
struct MyPlugin {}

#[async_trait]
impl Plugin for MyPlugin {
    fn id(&self) -> &'static str {
        "MyPlugin"
    }

    fn filter(&self, event: &Arc<Event>) -> bool {
        // Быстрая проверка: нужно ли обрабатывать событие
        matches!(event.data, EventType::Posted(_))
    }

    async fn process_event(&self, event: &Arc<Event>, config: &Arc<Configuration>) {
        // Обработка события
        if let EventType::Posted(posted) = &event.data {
            // Работа с сообщением
        }
    }
}
```

3. Добавить плагин в бота:
```rust
let mut bot = Bot::new(config);
bot.add_plugin(MyPlugin {});
bot.run().await;
```

### Работа с Mattermost API

Все API вызовы доступны через модуль `mattermost_api::apis`:

```rust
use mattermost_api::apis::{posts_api, channels_api};

// Получить посты канала
let posts = posts_api::get_posts_for_channel(
    &config,
    "channel_id",
    None,  // page
    Some(100),  // per_page
    None,  // since
    None,  // before
    None,  // after
    None,  // include_deleted
).await?;

// Создать пост
let new_post = mattermost_api::models::post::Post {
    channel_id: Some("channel_id".to_string()),
    message: Some("Hello!".to_string()),
    ..Default::default()
};
let created = posts_api::create_post(&config, new_post).await?;
```

### Регенерация mattermost-api

См. `/notes/generate_mm_api.md` для инструкций по регенерации API клиента из новой OpenAPI схемы.

**Важные шаги**:
1. Скачать новую `openapi.json` из Mattermost
2. Запустить генератор OpenAPI
3. Применить патчи для `bool` полей (regex замена в моделях)
4. Закомментировать проблемные API endpoints при необходимости

## Конфигурация

### Создание Bot configuration

```rust
use mattermost_api::apis::configuration::Configuration;

let mut config = Configuration::default();
config.base_path = "http://your-mattermost-server:8065".to_string();
config.bearer_access_token = Some("your_bot_token".to_string());
```

### Получение bot token

1. Создать бота в Mattermost (Integrations → Bot Accounts)
2. Скопировать Access Token
3. Добавить бота в нужные каналы/команды

## Рекомендации по разработке

### Логирование

Проект использует `tracing`:
- `tracing::debug!` - детальная информация (например, сырые WebSocket сообщения)
- `tracing::info!` - важные события (например, успешно распарсенное событие)
- `tracing::error!` - ошибки (например, неудачная десериализация)

### Обработка ошибок

- WebSocket подключение автоматически переподключается при ошибках
- Ошибки парсинга событий логируются, но не прерывают работу бота
- Каждый плагин обрабатывается независимо (ошибка в одном не влияет на другие)

### Производительность

- События передаются через `Arc<Event>` - избегайте клонирования
- `filter()` должен быть быстрым - сложная логика в `process_event()`
- `process_event()` вызывается параллельно для разных плагинов
- Используйте async/await для I/O операций

### Безопасность

- Никогда не коммитьте токены в git
- Используйте переменные окружения для конфиденциальных данных
- Валидируйте входные данные из событий
